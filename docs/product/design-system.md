# AI Life OS — Design System Bible

## Purpose

Этот документ описывает не цвета и кнопки, а философию интерфейса. Каждый экран должен уменьшать когнитивную нагрузку и помогать пользователю за несколько секунд понять состояние своей жизни.

## Design Philosophy

Интерфейс вдохновлен HUD из игр, а не классическими дашбордами. Минимум отвлекающих элементов. Максимум ясности. Пользователь приходит не 'поработать с приложением', а понять себя.

## Core Principles

1. Спокойствие важнее вау-эффекта.
2. Один экран лучше пяти вкладок.
3. Показывать главное, скрывать второстепенное.
4. Любая анимация должна объяснять изменение состояния.
5. Любой элемент обязан иметь причину существования.

## Visual Language

Использовать чистые поверхности, большие отступы, простые формы, минимум декоративных элементов. Цвета несут смысл: рост, внимание, риск, нейтральность. Никогда не использовать цвет только ради красоты.

## Life HUD

Главный экран содержит характеристики жизни, баланс, главную цель, три важнейших действия, ежедневный инсайт и краткую финансовую сводку. Пользователь должен понимать свое состояние за 5 секунд.

## Components

Карточки характеристик, прогресс целей, лента памяти, AI-инсайт, блок баланса, финансовый прогресс, карточка проекта, карточка привычки, строка быстрого ввода. Все компоненты имеют единый язык отступов и типографики.

## Interaction

Основной сценарий — одна строка ввода. Пользователь пишет естественным языком. Интерфейс не заставляет выбирать категории; ИИ делает это сам.

## Motion

Анимации медленные и спокойные. Изменения статусов плавно подчеркиваются. Не использовать анимации ради развлечения. Они должны поддерживать понимание происходящего.

## Typography

Крупные заголовки для состояния. Вторичный текст максимально лаконичный. Цифры характеристик и прогресс должны считываться быстрее текста.

## Accessibility

Высокий контраст, крупные зоны нажатия, управление с клавиатуры, поддержка экранных дикторов, отсутствие критичной информации, передаваемой только цветом.

## Tone of Interface

Интерфейс не обвиняет пользователя. Вместо 'Вы не выполнили задачи' — 'Сегодня был сложный день. Завтра продолжим.'

## Empty States

Пустые экраны обучают. Вместо 'Нет данных' показываются примеры того, что можно записать и какую ценность пользователь получит.

## Design Tokens

Все цвета, размеры, радиусы, тени, типографика и интервалы должны определяться через дизайн-токены. Запрещено использовать магические значения.

## Consistency Rules

Одинаковые действия выглядят одинаково. Один термин — одно значение. Любое изменение компонента отражается во всей системе.

## Definition of Done

Новый экран считается готовым только если: соответствует Manifesto, уменьшает когнитивную нагрузку, объясним пользователю без обучения и ощущается частью единой Life OS.
