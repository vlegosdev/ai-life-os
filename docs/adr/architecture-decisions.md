# AI Life OS — Architecture Decision Records

# Назначение ADR

ADR фиксирует не только что было выбрано, но и почему. Документ нужен, чтобы спустя полгода или несколько лет команда не повторяла старые споры и понимала ограничения каждого решения.

## Когда создавать ADR

изменяется граница доменного модуля

выбирается или заменяется ключевая технология

меняется схема хранения или модель памяти

появляется новый внешний провайдер

решение затрагивает приватность, безопасность или автономность AI

появляется необратимое или дорогое в миграции решение

# ADR-001 — Memory First как ядро системы

## Контекст

AI Life OS должна сохранять жизненный контекст до того, как он превратится в задачу, расход, цель или инсайт.

## Решение

Сделать Memory Engine первичным слоем: raw capture сохраняется неизменно, а остальные сущности создаются как производные представления.

## Положительные последствия

Не теряется исходный смысл пользователя.

Можно переинтерпретировать данные при улучшении моделей.

Проще строить долгосрочную память и provenance.

## Отрицательные последствия и риски

Усложняется модель данных.

Нужны версии и дедупликация производных сущностей.

Требуется строгая политика хранения персональных данных.

## Проверка решения

Решение пересматривается после появления реальных данных, инцидентов или существенного изменения масштаба. Метрики и условия пересмотра уточняются при реализации.

# ADR-013 — Минимальный foundation stack

## Статус
Accepted — 2026-07-12

## Контекст
Engineering Bible перечислял широкий целевой стек, а state-файлы утверждали, что стек еще не выбран. Это создавало прямое противоречие и позволяло агентам выбирать разные foundations.

## Решение
Для P0 утвержден минимальный baseline: pnpm workspaces + Turborepo, strict TypeScript, Next.js для web, Fastify для API, Zod для runtime validation, PostgreSQL как будущая основная БД и GitHub Actions для проверок. На P0 не внедряются аутентификация, мобильный клиент, pgvector, Redis, BullMQ, S3, OpenTelemetry, Sentry и LLM-интеграция.

## Последствия
Foundation становится однозначным и дешевым. Расширения добавляются отдельными задачами после появления измеренной необходимости.

# ADR-014 — Канонический roadmap и physical repository layout

## Статус
Accepted — 2026-07-12

## Контекст
В документации существовали две конкурирующие нумерации фаз и две разные физические структуры репозитория.

## Решение
Единственный канонический roadmap продуктовых фаз: `docs/product/development-roadmap.md`. Последовательности в Engineering Bible называются техническими вехами и не используются в project state. Единственный канонический physical layout: `apps/*` и `packages/*`; названия доменов в AI Engineering Bible являются концептуальной картой.

## Последствия
AI-агенты получают один источник для phase status и один layout для создания файлов.
