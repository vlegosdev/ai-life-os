# AI Life OS — Engineering Bible

# 0. Назначение документа

Engineering Bible фиксирует техническую правду проекта AI Life OS на момент начала разработки. Это не набор временных решений и не документация конкретного фреймворка. Документ определяет архитектурные границы, доменную модель, правила работы AI-компаньона, стандарты качества и порядок эволюции системы.

Как использовать через полгода
Перед добавлением крупной функции команда должна проверить: не дублирует ли она существующую сущность, не увеличивает ли когнитивную нагрузку, сохраняет ли объяснимость AI и соответствует ли принципу «человек принимает решение, система помогает».

# 1. Архитектурные принципы

## 1.1. Человек остается владельцем решений

Система предлагает, объясняет, прогнозирует и напоминает, но не принимает необратимые решения без подтверждения.

## 1.2. Один ввод — много полезных представлений

Пользователь пишет естественным языком. Система извлекает сущности, создает связи и отображает информацию в нужном контексте без ручной сортировки.

## 1.3. Память важнее чата

Чат — интерфейс. Основная ценность находится в структурированной, проверяемой и управляемой памяти.

## 1.4. Объяснимость по умолчанию

Любая оценка, совет или изменение показателя должны иметь понятное объяснение и ссылку на использованные факты.

## 1.5. Модульность и заменяемость

LLM, эмбеддинги, векторное хранилище, каналы уведомлений и внешние интеграции должны заменяться через адаптеры.

## 1.6. Privacy by design

Минимизация данных, шифрование, экспорт, удаление, локальные режимы и прозрачный журнал использования памяти проектируются до масштабирования.

## 1.7. Мягкое вмешательство

Хранитель внимания не блокирует и не стыдит. Он использует градуированное вмешательство: заметить → объяснить → предложить → запросить разрешение.

## 1.8. Стабильное ядро, изменяемая оболочка

Доменные сущности и события стабильны. Виджеты, экраны, формулы и персональные правила могут настраиваться пользователем без миграции всей системы.

# 5. Единая строка ввода и Capture Pipeline

## 5.1. Цель

Пользователь не выбирает заранее, что он вводит. Он пишет: «завтра записаться к стоматологу», «потратил 1800 на продукты», «хочу накопить миллион» или «идея: приложение как меню персонажа». Система должна принять ввод мгновенно, сохранить оригинал и затем безопасно интерпретировать.

## 5.2. Этапы обработки

Синхронно сохранить оригинальный текст как InboxItem.

Определить язык, источник, временные выражения и потенциально чувствительные данные.

Провести многоклассовую классификацию: task, goal, transaction, note, idea, journal, event, question или mixed.

Извлечь структурированные поля по JSON Schema.

Сопоставить сущности с существующим контекстом пользователя.

Оценить confidence и риск ошибочного действия.

Создать предложения изменений; автоматически применять только безопасные и обратимые операции.

Сгенерировать эмбеддинг и индексировать запись.

Запустить фоновые связи, резюме и возможные инсайты.

# 6. Memory Engine

## 6.1. Назначение памяти

Память должна не просто находить похожий текст, а сохранять историю человека, отделять факты от предположений, учитывать время, противоречия, значимость и право пользователя забыть данные.

## 6.2. Типы памяти

## 6.3. Модель MemoryItem

## 6.4. Извлечение памяти

Memory Retrieval формирует контекст в несколько стадий: фильтры безопасности и доступа → поиск по сущностям и времени → семантический поиск → ранжирование по актуальности, значимости и надежности → устранение дубликатов → упаковка в ограниченный контекст модели.

retrieval_score = 0.30 * semantic_similarity
                + 0.20 * entity_overlap
                + 0.15 * recency
                + 0.15 * salience
                + 0.10 * source_reliability
                + 0.10 * goal_relevance

## 6.5. Консолидация

ежедневное сжатие повторяющихся записей;

еженедельное обновление устойчивых предпочтений;

сохранение исходников и трассировки любого резюме;

новый факт не перезаписывает старый молча — создается версия или противоречие;

низкоуверенные выводы не становятся «правдой» без повторных подтверждений.

## 6.6. Право на забвение

Удаление должно очищать первичную запись, производные эмбеддинги, кеши, связи и будущие резюме. Для резервных копий действует документированный срок физического удаления. Пользователь может просмотреть, исправить, экспортировать или запретить использование любой памяти.


# Canonical MVP domain boundaries

These names are canonical in code, state, and documentation.

| Domain | Owns | Must not own |
|---|---|---|
| Capture | input validation and submission workflow | durable memory storage or AI interpretation |
| Memory | original entry, metadata, versions, retrieval | tasks, finance calculations, or free-form LLM decisions |
| Understanding | schema-validated AI proposals derived from Memory | overwriting original Memory or authoritative writes |
| Goals | goals, progress rules, and goal-related actions | raw capture and financial ledger truth |
| Finance | transactions, budgets, debts, scenarios, deterministic calculations | medical advice or free-form LLM arithmetic |
| Life Stats | explainable snapshots based on verified signals | judging personality or inventing evidence |
| Insights | evidence-backed observations and explanations | changing source records |
| Attention Guardian | overload signals and attention recommendations | blocking user actions without explicit permission |
| Companion | conversation, orchestration, and presentation | being a source of truth or taking irreversible action |

For the first vertical slice only Capture and Memory are active. Other domains remain conceptual until their roadmap phase.

# Canonical AI-to-code contract

The LLM receives a minimal, explicitly permitted context object and a task-specific JSON Schema. It returns a proposal with `type`, `payload`, `confidence`, `evidence_memory_ids`, and `explanation`. All output is untrusted. Application code validates schema, permissions, referenced records, numeric values, dates, and idempotency before any write.

The LLM may classify, summarize, suggest links, and draft actions. It may not directly write to the database, calculate authoritative money or scores, decide authorization, silently overwrite Memory, or perform irreversible external actions. Low-confidence proposals remain drafts and are editable or rejectable by the user.

## 8.2. Источники доказательств

## 8.3. Формула оценки

Каждый показатель — не оценка личности, а снимок состояния сферы. Он строится из нормализованных сигналов с персональными весами и уровнем уверенности.

raw_score = Σ(weight_i * normalized_signal_i) / Σ(weight_i)
smoothed_score_t = α * raw_score_t + (1-α) * score_(t-1)
confidence = coverage * source_quality * consistency
shown_score = round(smoothed_score) with confidence band

## 8.4. Баланс

Баланс не означает, что все сферы всегда равны. Он показывает перекос относительно личных целевых диапазонов. В период запуска проекта карьера может быть выше нормы, но система должна показать цену этого решения и длительность перекоса.

imbalance = weighted_stddev((score_i - target_i) / tolerance_i)
balance_score = clamp(100 - k * imbalance, 0, 100)

## 9.2. Goal

Разбить цель на этапы.

Оценить сценарии: базовый, оптимистичный, стрессовый.

Предложить ближайшее обратимое действие.

Получить подтверждение пользователя.

Регулярно пересматривать план при изменении условий.

## 9.4. Приоритизация дня

priority = goal_impact * urgency * leverage * feasibility
           - energy_cost * overload_penalty
           - balance_risk

## 10.3. Прогноз денежных целей

monthly_free_cash = verified_income - essential_expenses - debt_minimums
months_to_target = ceil((target - current_saved) / planned_monthly_contribution)

Прогноз обязан показывать допущения. Для квартиры система должна отдельно показать первоначальный взнос, долговую нагрузку, ежемесячный платеж, резерв после сделки и чувствительность к ставке.

## 10.4. Приоритет закрытия долгов

Поддерживаются стратегии avalanche, snowball и пользовательская. Любая рекомендация содержит сравнение полной переплаты, срока, психологической сложности и обязательных платежей.

## 10.5. Категоризация и качество данных

правила пользователя имеют приоритет над AI;

переклассификация обучает персональное правило;

переводы между счетами не считаются расходом;

возвраты связываются с исходной операцией;

неподтвержденные транзакции не влияют на официальные прогнозы;

покрытие данных показывается рядом с финансовой оценкой.

## 10.6. Финансовая безопасность

банковские токены хранятся в secrets vault;

реквизиты и полные номера карт не отправляются в LLM;

write-доступ к банку отсутствует в ранних версиях;

любые платежные действия требуют отдельного подтверждения и сильной аутентификации.

# 12. Insights Engine

## 12.1. Виды инсайтов

описательные: что изменилось;

диагностические: какие факторы связаны с изменением;

# 15. Хранение данных

## 15.1. PostgreSQL

Транзакционные сущности, связи, настройки, события, аудиты и версии хранятся в PostgreSQL. JSONB допустим для изменчивых метаданных, но ключевые поля, используемые в фильтрах и ограничениях, должны быть нормализованы.

## 15.2. Vector index

pgvector хранит эмбеддинги MemoryItem, ConversationChunk и SourceDocument. Индекс является производным: его можно перестроить из первичных данных. Версия embedding-модели хранится рядом с вектором.

## 15.3. Object storage

Вложения шифруются, проверяются антивирусом, имеют content hash и lifecycle policy. Извлеченный текст хранится отдельно с ссылкой на исходный объект.

# 16. Безопасность и приватность

## 16.1. Модель угроз

утечка базы или резервной копии;

компрометация LLM-провайдера;

prompt injection из импортированного контента;

чрезмерные полномочия агента;

ошибочная выдача памяти другого пользователя;

злоупотребление уведомлениями;

инсайдерский доступ;

кража устройства или сессии.

## 16.2. Контрмеры

строгая tenant isolation на уровне запросов и тестов;

encryption in transit и at rest;

отдельные ключи/секреты и ротация;

редакция PII перед внешней моделью, когда возможно;

sandbox для обработки файлов и контента;

tool allowlist и scope-based authorization;

step-up authentication для экспорта и критических действий;

полный audit trail чтения чувствительных данных;

регулярные security reviews и dependency scanning.

## 16.3. Prompt injection

Текст из веб-страницы, письма или документа рассматривается как недоверенные данные, а не инструкция. Он помещается в маркированный контейнер, не может изменять системную политику и не получает доступ к инструментам. Любой tool call проверяется сервером независимо от текста модели.

## 16.4. Согласие

Каждая интеграция и чувствительная категория памяти имеют отдельное согласие. Система показывает, какие данные используются для конкретного совета. Согласие можно отозвать без удаления всего аккаунта.

## 17.2. Graceful degradation

если LLM недоступна, ввод сохраняется в Inbox;

если vector search недоступен, работает поиск по сущностям и тексту;

если интеграция сломана, показывается дата последней синхронизации;

HUD использует последний подтвержденный snapshot и маркирует устаревание;

фоновые задачи повторяются с backoff и dead-letter queue.

## 17.3. Идемпотентность

Все команды создания из мобильного клиента и интеграций используют idempotency key. Повторный webhook или сетевой retry не должен создавать двойной расход, задачу или платеж.

# 23. Стоимость AI и производительность

## 23.1. Бюджетирование

малые модели для классификации и извлечения;

кеширование стабильных резюме;

асинхронная консолидация;

лимит контекста через retrieval и summaries;

per-user и per-feature cost attribution.

## 23.2. Контроль контекста

В сильную модель отправляется только минимальный релевантный контекст. Финансовые таблицы агрегируются сервером; модель получает итоги и допущения, а не тысячи транзакций. Цифры рассчитываются детерминированным кодом.

## 23.3. Rate limits

Лимиты зависят от тарифа и риска. Критические пользовательские операции не блокируются из-за лимита генераций: capture всегда сохраняется, а AI-обработка может быть отложена.

# 24. Этика и границы продукта

## 24.1. Медицинские и психологические темы

Система может отражать данные, помогать подготовить вопросы врачу и поддерживать здоровые рутины. Она не ставит диагноз, не назначает лечение и не создает ложную уверенность. Сигналы высокого риска обрабатываются отдельной safety policy.

## 24.2. Финансовые темы

Система объясняет бюджет, долги и сценарии. Инвестиционные и кредитные решения маркируются как образовательные расчеты с допущениями и требуют проверки актуальных условий.

## 24.3. Отношения

Система не оценивает людей «по баллам», не манипулирует социальными связями и не выдает категоричные психологические диагнозы. Она может напомнить о важных контактах и показать фактические паттерны.

## 24.4. Антизависимость

нет бесконечной ленты;

нет случайных наград;

нет искусственной срочности;

нет наказания за отсутствие;

цель продукта — освободить время, а не максимизировать экранные минуты.
