# AI Life OS — AI Engineering Bible

## Purpose

Этот документ определяет не только стиль кода, но и образ мышления инженера.
Любой человек или ИИ, работающий над AI Life OS, обязан следовать этим принципам.
Цель — сохранить целостность продукта спустя годы.

## Engineering Philosophy

Мы оптимизируем не скорость написания кода, а качество архитектуры.
Каждая новая функция должна соответствовать Manifesto, Product Vision и Engineering Bible.
Если реализация противоречит философии продукта — реализация неверна.

## Golden Rule

Сначала понять проблему.
Потом спроектировать решение.
Потом написать код.
Никогда наоборот.

## Role of AI

ИИ — инженер-помощник, а не архитектор.
Он предлагает варианты, пишет код, рефакторит, тестирует и документирует.
Архитектурные решения принимает человек.

## Architecture Principles

Feature-first, Domain Driven Design, Clean Architecture, строгая типизация,
объяснимость решений, модульность, низкая связанность, высокая связность внутри модуля.

## Repository Structure

### Canonical physical layout

```text
apps/
  web/
  api/
packages/
  config/
  domain/
  database/
  ui/
  ai/
docs/
ai/
infra/
tests/
```

This `apps/*` + `packages/*` layout is canonical. The names `memory`, `finance`, `goals`, `projects`, `stats` and `design-system` describe conceptual domain boundaries, not mandatory top-level folders. Create physical packages for them only when implemented scope and an ADR justify the split.

## Domain Modules

Memory — единый источник истины.
Life Stats — объяснимые снимки состояния сфер жизни.
Goals — цели.
Projects — проекты.
Tasks — действия.
Finance — деньги.
Insights — выводы.
Attention Guardian — защита внимания и рекомендации при перегрузке.
Companion — поведение ИИ.

## Memory Engine Rules

Любой ввод сначала сохраняется как Memory.
Только затем классифицируется.
Никогда не терять исходную запись пользователя.
Все связи между объектами должны быть воспроизводимыми.

## LLM Usage

LLM используется только там, где необходимы понимание языка,
классификация, суммаризация, генерация инсайтов и диалог.
Все детерминированные вычисления выполняются обычным кодом.

## Prompt Rules

Промпты хранятся версионируемо.
Любое изменение промпта сопровождается описанием причины.
Промпты не содержат бизнес-логики, только инструкции модели.

## Explainability

Любой AI-вывод должен иметь объяснение:
какие данные использованы,
почему сделан вывод,
насколько он уверен.

## Cursor Rules

Cursor запрещено:
• менять архитектуру без запроса;
• использовать any;
• смешивать UI и бизнес-логику;
• создавать компоненты >300 строк;
• создавать дублирующий код;
• переименовывать доменные сущности без ADR.

## Code Style

TypeScript strict.
React — функциональные компоненты.
Маленькие файлы.
Чистые функции.
Никакой скрытой магии.
Комментарии только для объяснения решений, а не очевидного кода.

## Testing

Юнит-тесты для бизнес-логики.
Интеграционные тесты для API.
Golden datasets для AI.
Snapshot запрещены там, где важна логика.

## Security

Шифрование персональных данных.
Минимизация хранения секретов.
Защита от prompt injection.
Проверка прав доступа перед любым AI-действием.

## Performance

Lazy loading.
Background jobs.
Кэширование embeddings.
Наблюдаемость через tracing и метрики стоимости AI.

## Git Workflow

main всегда стабилен.
Feature branches.
Conventional commits.
Обязательный code review.
ADR для крупных архитектурных изменений.

## Definition of Done

Код соответствует всем Bible.
Есть тесты.
Есть документация.
Нет деградации UX.
Нет увеличения когнитивной нагрузки пользователя.

## AI Review Checklist

Перед каждым merge ИИ обязан проверить:
✓ Соответствие Manifesto
✓ Соответствие Product Vision
✓ Соответствие Engineering Bible
✓ Объяснимость
✓ Производительность
✓ Безопасность
✓ Пользователь получает больше ценности, а не больше сложности.

## Project Constitution

1. Пользователь важнее метрик.
2. Внимание важнее вовлеченности.
3. Баланс важнее экстремума.
4. Память важнее заметок.
5. ИИ помогает, а не управляет.
6. Простота важнее количества функций.
7. Архитектура важнее скорости.
8. Код обязан быть понятным через пять лет.

## Long-term Vision

Через несколько лет любой новый разработчик должен открыть этот документ,
прочитать его за вечер и начать писать код так, словно участвовал в проекте с первого дня.
Это главный критерий качества AI Engineering Bible.
